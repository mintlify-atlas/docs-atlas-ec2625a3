## Overview

Cheat Sheets follows a feature-first architecture with clean separation of concerns, utilizing Flutter best practices and modern state management patterns.

## Feature Architecture

Each feature follows a clean architecture pattern with three distinct layers:

<Steps>
  <Step title="Domain Layer">
    Contains business logic and models:
    - Pure Dart models (no framework dependencies)
    - Domain entities and value objects
    - Uses `freezed` for immutable data classes
    
    Example: `features/cheatsheets/domain/cheatsheet.dart`
  </Step>

  <Step title="Data Layer">
    Handles data sources and repositories:
    - **Data Sources**: API clients, local storage
      - `remote_cheatsheet_data_source.dart` - API calls
      - `asset_cheatsheet_data_source.dart` - Local assets
    - **Repositories**: Abstract data access
      - `cheatsheet_repository.dart` - Interface
      - `cheatsheet_repository_impl.dart` - Implementation
    - Uses `fpdart` for functional error handling
    
    <Note>
      The data layer uses the `Either` type from `fpdart` to handle errors functionally, returning `Either<Exception, Data>` instead of throwing exceptions.
    </Note>
  </Step>

  <Step title="Presentation Layer">
    UI components and state management:
    - **Providers**: Riverpod state management
    - **Screens**: Full-page UI components
    - **Widgets**: Reusable UI components
    
    Example structure:
    ```
    presentation/
    ├── providers/
    │   ├── cheatsheet_providers.dart
    │   └── cheatsheet_search_controller.dart
    ├── screens/
    │   ├── cheatsheet_list_screen.dart
    │   ├── cheatsheet_detail_screen.dart
    │   └── cheatsheet_search_screen.dart
    └── widgets/
        └── custom_list_tile.dart
    ```
  </Step>
</Steps>

## State Management

The app uses **Riverpod 2.6+** with code generation for type-safe state management.

### Key Concepts

<Accordion title="Riverpod Providers">
  Providers are the core of state management:
  
  ```dart
  // Code-generated provider
  @riverpod
  class CheatsheetList extends _$CheatsheetList {
    @override
    Future<List<CheatsheetMeta>> build() async {
      // Fetch data
    }
  }
  ```
  
  The app uses:
  - `@riverpod` annotation for code generation
  - `riverpod_generator` for boilerplate reduction
  - `hooks_riverpod` for widget integration
</Accordion>

<Accordion title="Provider Types">
  Different providers for different use cases:
  
  - **FutureProvider**: Async data fetching
  - **StateProvider**: Simple mutable state
  - **NotifierProvider**: Complex state logic
  - **StreamProvider**: Real-time data streams
  
  See `lib/src/features/cheatsheets/presentation/providers/` for examples.
</Accordion>

<Accordion title="Dependency Injection">
  Riverpod handles dependency injection:
  
  ```dart
  @riverpod
  CheatsheetRepository cheatsheetRepository(CheatsheetRepositoryRef ref) {
    final dataSource = ref.watch(cheatsheetDataSourceProvider);
    return CheatsheetRepositoryImpl(dataSource: dataSource);
  }
  ```
  
  This creates a dependency graph that's automatically managed.
</Accordion>

## Routing & Navigation

The app uses **GoRouter 15.1+** for declarative routing.

### Router Configuration

<CodeGroup>
```dart app_router.dart
final _rootNavigatorKey = GlobalKey<NavigatorState>();

final appRouter = GoRouter(
  initialLocation: "/cheatsheet",
  navigatorKey: _rootNavigatorKey,
  routes: $appRoutes, // Generated routes
);
```

```dart app_routes.dart
// Uses go_router_builder for type-safe routes
@TypedGoRoute<CheatsheetRoute>(
  path: '/cheatsheet',
  routes: [
    TypedGoRoute<CheatsheetDetailRoute>(
      path: ':id',
    ),
  ],
)
class CheatsheetRoute extends GoRouteData {
  @override
  Widget build(BuildContext context, GoRouterState state) {
    return const CheatsheetScreen();
  }
}
```
</CodeGroup>

<Tip>
  The app uses `go_router_builder` for code generation, providing type-safe navigation with compile-time route validation.
</Tip>

## Data Flow

The typical data flow in the app:

<Steps>
  <Step title="User Interaction">
    User interacts with a widget (e.g., taps a cheat sheet)
  </Step>

  <Step title="Provider Notification">
    Widget calls a provider method or navigates via GoRouter
  </Step>

  <Step title="Repository Layer">
    Provider calls repository which returns `Either<Exception, Data>`
  </Step>

  <Step title="Data Source">
    Repository delegates to data source (remote API or local assets)
  </Step>

  <Step title="Error Handling">
    Errors are mapped to app-specific exceptions:
    ```dart
    TaskEither<AppException, List<CheatsheetMeta>> list() {
      return _dataSource.list()
        .mapLeft((error) => error.toAppException());
    }
    ```
  </Step>

  <Step title="UI Update">
    Riverpod notifies widgets, which rebuild with new data
  </Step>
</Steps>

## Code Generation

The app heavily relies on code generation for:

1. **Riverpod Providers** (`@riverpod`)
2. **GoRouter Routes** (`@TypedGoRoute`)
3. **Freezed Models** (`@freezed`)
4. **JSON Serialization** (`@JsonSerializable`)
5. **Environment Variables** (`@Envied`)

<Warning>
  Always run `dart run build_runner build` after:
  - Adding new providers
  - Modifying routes
  - Changing model classes
  - Updating `.env` files
</Warning>
